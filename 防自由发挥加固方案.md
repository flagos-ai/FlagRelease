# 防 Claude 自由发挥 —— 编排层加固方案（待 review，未实施）

## 背景
分支 B 流程中，凡"决定流程走向"的判断若只靠 prompt 的"必须/禁止"约束、缺产物级闸门，Claude 偶发会自作主张跳过规定脚本。已有可靠范式两种：产物校验闸门（run_eval_if_missing / step7_gate.py）、越界回滚。本方案把三个尚未覆盖的口子补齐，全部复用现有范式。

## 闸门覆盖现状
| 环节 | 现状 | 风险 |
|---|---|---|
| 步骤4 V1精度 | ✅ run_eval_if_missing | 低 |
| 步骤4 V2精度 | ⚠ 无独立兜底 | 中 |

## 口子1：V1 三选强制闸门（最高优先，上游根因）

**问题**：baseline_selector.py 在 run_pipeline.sh 中无任何 shell 强制调用点，完全靠 Claude "记得"按 CLAUDE.md 调用。Claude 可能自起服务测一下就断言"V1可跑/不可跑"，导致 baseline.v1_variant / v1_available 无确定性来源。这是 V1.3 误判、vendor+fl 衔接等问题的上游——三选没被强制走，后面全是空中楼阁。

**判据（agent 难伪造）**：baseline_selector.py 落盘 v1_baseline_selection.json，含 attempts[] 数组（每个变体的 service_ok/smoke_passed/smoke_answer 记录）。Claude 手写 context 不会产生这个结构完整的文件。

**方案**：仅分支 B，段1后 / 段2前加校验：
1. 读 context baseline.v1_variant 是否 ∈ {v1.1,v1.2,v1.3,none}
2. 且 results/v1_baseline_selection.json 存在、含 attempts[] 且每项有 smoke_passed 字段
3. 二者任一不满足 → shell 直接 docker exec 调 baseline_selector.py（--vendor-plugin auto --json），产出后同步 context
4. 落盘一道 v1_gate.py（仿 step7_gate.py），输出 ok/needed，shell case 分派

**注意**：baseline_selector 会启停服务，执行有成本；需放在服务已就绪、精度评测之前的窗口。

## 口子2：V2 精度评测兜底（中优先，零成本）

**问题**：run_eval_if_missing 已是通用参数化函数，但只调用一次（gpqa_native.json = V1）。V2 精度 gpqa_flagos.json 未兜底。

**方案**：run_eval_if_missing 已支持传 OUTPUT_FILE，直接加一次调用：
```
run_eval_if_missing "gpqa_flagos.json" "${SEG2_START_ISO}" || true
```
**关键差异**：V2 评测要求 FlagGems 服务在跑（USE_FLAGGEMS=1），与 V1 native 服务不同。现有 run_eval_if_missing 只 curl 检查服务存活（HTTP 200）不校验是哪种模式。需在 V2 兜底前确认当前服务是 FlagGems 模式（读 context 的 service 状态或 /etc/environment 的 USE_FLAGGEMS），否则可能拿 native 服务测出错误的 V2 分数。若模式不符则先切服务再测——这一步不能省，否则兜底会产生污染数据，比不兜底更糟。

## 口子3：发布镜像真存在校验（高价值，不可逆环节）

**问题**：发布已有 --only-harbor shell 兜底，但成功判定 = main.py 退出码 + 解析 [RELEASE_SUMMARY]。若 push 实际失败但脚本误报成功，流程当成功继续。发布对外不可逆，最该有独立产物校验。

**约束**：环境无 skopeo。用 docker manifest inspect docker://<harbor_tag> 或 Harbor API（/api/v2.0/projects/<proj>/repositories/<repo>/artifacts/<tag>）。凭证：复用 harbor_user/harbor_pass（脚本已有）。

**方案**：发布段（V2/V3/V4/V5 各发布点）后加校验：
1. 从 context 取本次应发布的 harbor tag（registry_url / harbor_tag）
2. docker manifest inspect <tag>（需先 docker login harbor）确认 manifest 存在
3. 存在 → 标记发布成功；不存在 → 重试一次 --only-harbor，仍失败则明确报错写 context（不静默当成功）

**注意**：V1.3 双 tag（--also-tag v3）场景要校验两个 tag 都存在。docker manifest inspect 对私有 registry 需 experimental 或已登录，需先确认 docker 版本支持。

3. 口子3（价值最高但需先确认 docker manifest inspect 在目标环境可用 + Harbor 登录时序）

## 共同原则
所有新闸门遵循 step7_gate.py 范式：判据只用 agent 无法伪造的产物事实（文件结构/时间戳/registry manifest），不读 Claude 写的 ledger/ok 字段；不满足则 shell 直接执行规定脚本兜底，不再交回 Claude 决策。
